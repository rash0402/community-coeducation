# 東京電機大学工学部第二部 地域共育プログラム プロジェクト定義書

> [!ABSTRACT] プロジェクト概要
>
> **ミッション:** 「教室で理論を、地域企業で実践力を」
>
> **コア・コンセプト:** 大学が「仲介」するのではなく、**「信頼できる情報の場」と「学生の質保証（認定）」のみを提供する「プラットフォーム型（掲示板方式）」**の産学連携モデル。
>
> **目的:**
> 1. 夜間部学生への「経済的支援（生活費）」と「キャリア形成（技術習得）」の両立
> 2. 地域中小企業の「若手人材不足・技術継承問題」の解消

---

## 1. 運用スキームとルール

### 1.1 基本フロー（ダイレクト・リクルーティング方式）

> [!NOTE] 運用方針
> 事務局（大学）は「情報のゲートキーパー」に徹し、個別のマッチングや面接調整は行わない。

```
【企業】求人登録フォームで申請
        ↓
【事務局】掲載基準チェック（自動 + 目視）
        ↓
【事務局】求人リスト（スプレッドシート）を学内公開
        ↓
【学生】オンデマンド研修受講 → テスト合格
        ↓
【システム】認定ID自動発行 + 合格メール送信
        ↓
【学生】認定IDを持って企業に直接応募
        ↓
【企業】検証ページで認定IDを確認
        ↓
【企業 ⇔ 学生】雇用契約を直接締結
```

### 1.2 重要ルール（Quality & Risk Control）

| 項目 | ルール内容 | 目的 |
|------|------------|------|
| **契約期間** | 「学期末（3月31日 / 9月30日）」区切りでの契約更新を推奨 | 学業（時間割変更）優先のため、半期ごとにシフトや契約を見直せるようにする |
| **掲載期限** | 【自動失効方式】求人情報の掲載期間は「当該学期の末日」まで。継続希望の場合は確認メールで更新手続き | 「空求人」トラブルを防止し、情報の鮮度を保つ |
| **移動時間** | フォーム登録時に「大学までの移動時間」を申告させ、18:00講義開始に間に合わない求人は排除 | 遅刻常習化の防止。物理的に通学可能な企業のみに限定 |
| **認定確認** | 【認定ID + 検証システム方式】テスト合格時にユニークな認定IDを発行。企業はオンライン検証ページでリアルタイム確認 | 認定証偽造防止、処分（停止/取消）の即時反映 |
| **成績要件** | 【自己誓約方式】認定申請時に「退学勧告等を受けていないこと」への誓約チェックを必須化（事務局によるGPA確認は行わない） | 確認負荷ゼロ化。学業優先の原則を学生自身の責任として約束させる |
| **保険適用** | 【企業責任の原則】本件は「雇用契約」であるため、労災保険の適用を企業の義務とする。業務中の損害賠償リスクも原則企業負担（大学の学研災は適用外） | 賠償リスクの回避。「学生が機械を壊した」等のトラブル時に、大学や学生個人に請求がいかないよう予め線引き |

## 2. 認定証検証システム

### 2.1 概要

従来のメール認定証のみでは「偽造リスク」「処分の強制力不足」という課題があった。これを解決するため、**認定ID + オンライン検証システム**を導入する。

### 2.2 認定ID仕様

```
TDU-YYYYT-NNNNN

TDU   : 固定プレフィックス（Tokyo Denki University）
YYYY  : 認定年度（西暦4桁）
T     : 学期（S=春学期, F=秋学期）
NNNNN : 連番（5桁、ゼロ埋め）
```

**例:**
- `TDU-2025S-00001` : 2025年春学期の1人目
- `TDU-2025F-00042` : 2025年秋学期の42人目

### 2.3 システム構成

```
┌─────────────────────────────────────────────────────────────┐
│                    Google Workspace                         │
├─────────────────────────────────────────────────────────────┤
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐  │
│  │ Google Forms │───▶│ スプレッドシート │◀───│ Google Apps  │  │
│  │ (認定テスト) │    │  (認定者DB)   │    │   Script     │  │
│  └──────────────┘    └───────┬──────┘    └──────────────┘  │
│                              │                              │
│                              ▼                              │
│                    ┌──────────────┐                         │
│                    │ Google Sites │                         │
│                    │ (検証ページ) │                         │
│                    └──────────────┘                         │
└──────────────────────────────────────────────────────────────┘
```

### 2.4 認定者データベース（スプレッドシート）

| 列 | 項目名 | 説明 |
|:--:|--------|------|
| A | 認定ID | `TDU-YYYYT-NNNNN`形式 |
| B | 学籍番号 | 学内システムの学籍番号 |
| C | 氏名 | 学生氏名（検証ページには非表示） |
| D | メールアドレス | 大学メール |
| E | 認定日 | テスト合格日 |
| F | 有効期限 | 認定日から1年後（自動計算） |
| G | ステータス | `有効` / `停止` / `取消` / `期限切れ` |
| H | ステータス更新日 | 最終更新日 |
| I | 備考 | 取消理由など |

### 2.5 ステータス定義と検証結果

| ステータス | 説明 | 検証ページ表示 | 採用可否 |
|------------|------|----------------|:--------:|
| **有効** | 認定資格あり | ✅ 認定有効 | ○ |
| **停止** | Lv.2処分（1学期間） | ⚠️ 資格停止中 | × |
| **取消** | Lv.3処分（永久） | ❌ 資格取消済み | × |
| **期限切れ** | 有効期限超過 | ⏰ 有効期限切れ | × |

### 2.6 検証ページURL

```
https://sites.google.com/dendai.ac.jp/kyoiku-ninsho/verify
```

> [!IMPORTANT] 企業への周知事項
> - **「認定有効」以外の学生は採用しないでください**
> - 本プログラムの求人に応募できるのは認定学生のみ
> - 確認せずに採用した場合、トラブル時に大学のサポートを受けられない場合あり

## 3. アプローチ方向と個人情報保護（Security Policy）

> [!DANGER] セキュリティ基本方針
> **「持たざる防御」**を徹底する。外部（企業）からアクセス可能な学生データベースは構築せず、攻撃対象領域を極小化する。

### 3.1 アプローチの限定（逆オファーの禁止）

- **Pull型限定:** 本システムは「学生から企業への応募」のみ
- **情報の階層化（袋とじ方式）:**
  - **Layer 1（公開）:** 企業名、業務内容、条件のみ掲載（連絡先非公開）。全学生が閲覧可
  - **Layer 2（限定）:** 電話番号・担当者名を含む完全情報。研修合格メール内にアクセス権（URL/PW）を記載
- **分散管理:** 応募用履歴書は学生から企業へ直接提出させ、大学サーバーに詳細な個人情報を蓄積しない

### 3.2 運用上の脆弱性対策（Cyber Hygiene）

| 脅威 | 対策 |
|------|------|
| 求人詐欺（Resume Harvesting） | 初回登録企業は「国税庁 法人番号公表サイト」にて実在性を確認 |
| マルウェア（Emotet等） | 成績確認を「自己誓約」へ変更し、ファイルのアップロード・ダウンロード業務を廃止 |
| 設定ミス（Cloud Misconfig） | フォーム・回答データの共有設定は「特定ユーザー（職員）のみ」。「リンクを知っている全員」設定を禁止 |
| 総当たり攻撃 | 検証ページで連続10回の検証失敗時にIP一時ブロック |
| 情報漏洩 | 検証結果に氏名・学籍番号は表示しない |

## 4. リスク管理体制（トリアージ方式）

トラブル対応を「深刻度」で分類し、担当部署の責任範囲と負荷を明確化する。

> [!WARNING] トラブルシューティング基準
> 大学は原則として「労働契約」には介入しないが、「教育機関」としての懲戒権（参加停止・取消）を行使する。

### 4.1 トリアージレベル定義

| レベル | 事象例 | 対応方針 | 担当 | システム処理 |
|:------:|--------|----------|------|--------------|
| **Lv.1** | 遅刻、欠勤、業務ミス | **不介入（当事者解決）** | 企業 ⇔ 学生 | なし |
| **Lv.2** | 継続的な態度不良、無断欠勤（3日以上）、掲載基準違反の常習 | **認定停止（1学期間）** | 事務局 | ステータス → `停止` |
| **Lv.3** | ハラスメント、労災隠し、賃金未払い、重大な契約違反 | **認定取消（永久）** | 専門部署 | ステータス → `取消` |

### 4.2 学生の資格停止と再認定

| 区分 | 処分 | 再認定の可否 |
|------|------|--------------|
| **Lv.2 参加停止** | 1学期間（6ヶ月間）の参加資格停止 | 停止期間終了後、反省レポートと事務局面談を経て再認定可能（新規IDで再発行） |
| **Lv.3 永久排除** | プロジェクト参加資格の永久剥奪 | **再認定不可** |

### 4.3 提携企業の資格停止と再認定

| 区分 | 事由例 | 処分 | 再認定の可否 |
|------|--------|------|--------------|
| **Lv.2 掲載停止** | 掲載基準不適合（移動時間60分超等）が再三見られた場合。または、学生のLv.2が頻繁に発生 | 1学期間（6ヶ月間）の掲載停止 | 改善計画書の提出と事務局面談を経て再認定可能 |
| **Lv.3 永久排除** | ハラスメント、賃金未払い、重大な契約違反（労災不適用など）、反社会的勢力との関与 | プロジェクト提携資格の永久剥奪 | **再認定不可** |

### 4.4 処分の効力

- ステータス変更は**即時**に検証ページへ反映
- 企業が検証ページで学生の認定IDを確認すると、最新のステータスが表示される
- これにより、停止・取消された学生が新たな企業に採用されることを防止

### 4.5 エスカレーション先

| 事象 | エスカレーション先 |
|------|-------------------|
| ハラスメント相談 | 学生相談室 |
| 労災・怪我 | 学生厚生課 |
| 法的問題（賃金未払い等） | 総務部法務担当 |
| 判断に迷う場合 | 工学部第二部長 |

## 8. 残存する懸念・議論ポイント（Future Issues）

### 8.1 コールドスタート問題

- **課題:** サービス開始時に「求人0件」だと学生が来ず、「学生不在」だと企業が来ない
- **対策:** 五十嵐先生のリレーションによる**「モデル企業（5社程度）」の先行確保**を必須とする

### 8.2 学内管轄（セクショナリズム）への配慮

- **課題:** キャリアセンター（就職担当）や教務課（成績担当）からの反発リスク
- **対策:**
  - 本件は「就職斡旋」ではなく**「地域貢献活動」**と定義
  - 成績確認は学生の自己誓約で完結させ、教務課の手を煩わせない
  - 求人データはキャリアセンターにも共有し「協力体制」をとる

### 8.3 学生の応募率

- **課題:** 「認定研修（動画視聴＋テスト）」というハードルにより、初動が鈍くなる懸念
- **対策:** ポスター等で「コンビニバイトより割が良い（時給・経験・就活）」メリットを強力に訴求

### 8.4 契約期間の柔軟性

- **課題:** 「学期末（3月/9月）更新」推奨ルールに対し、繁忙期が異なる企業からの不満
- **対策:** 原則は学期末だが、フォーム上で「同意しない（個別事情あり）」も選択可とし、完全排除はしない（ただし強く推奨する）

### 8.5 応募ゼロへの対応（企業クレーム対策）

- **課題:** 「求人を出したが学生が来ない」という企業からの不満
- **対策:**
  1. **前提共有:** 規約にて「採用の確約ではない（マッチング保証なし）」ことを明記
  2. **自己診断ツールの提供:** 「時給はコンビニより高いか？」等のチェックリストを渡し、条件改善を促す
  3. **不介入:** 事務局による「個別の学生紹介」は行わず、市場原理に委ねるスタンスを維持

### 8.6 よくあるケースと対応指針（FAQ）

| ケース | 事象 | 対応 |
|--------|------|------|
| **業務内容の不一致** | 「技術職と聞いていたのに掃除ばかり」 | 5S（整理整頓）は技術の基本。3ヶ月経過しても改善されない場合は次回の契約終了（辞退）を推奨 |
| **早期離職（バックレ）** | 「学生が連絡なしに来なくなった」 | 企業には就業規則に基づく処分を依頼。大学側は当該学生の認定証を即時剥奪 |
| **掲載拒否への抗議** | 「なぜウチの求人を載せない！」 | 「18:00の講義開始」という絶対基準を提示し、マップでの客観的な所要時間を示して断る。例外は認めない |
| **認定落ちへの不満** | 「働きたいのに認定されないのはおかしい」 | 「本プロジェクトは学業との両立を最優先としているため」と回答 |
| **実績ゼロ企業からの不安** | 「電大生の採用実績がないため応募が来ないのでは」 | 「第一期生として歓迎します」といったポジティブなメッセージを添えるようアドバイス |

## 9. ペルソナ分析手法

施策検討やリスク評価において、16名のペルソナを活用した分析手法を定義している。

### 9.1 マトリクス分析

施策に対する各ペルソナの反応（賛成/中立/反対）を一覧化し、賛否の分布と潜在的な問題点を可視化する。

**判定基準:**
- 反対が3名以上 → 施策の修正または追加対策を検討
- 特定カテゴリで反対多数 → そのカテゴリ向けの補完策を検討

### 9.2 重み付けスコアリング

ペルソナごとの影響度を数値化（重み合計: 35）し、施策の優先順位付けを定量的に行う。

| 合計スコア | 判定 |
|:----------:|------|
| +20以上 | 強く推奨 |
| +10〜+19 | 推奨（軽微な調整で実施） |
| 0〜+9 | 要検討（反対層への対策必要） |
| -1〜-10 | 慎重（大幅な修正が必要） |
| -11以下 | 見送り（根本的な見直し） |

### 9.3 対立ペア討論

対立する利害を持つペルソナ同士で仮想討論を行い、妥協点や新たな解決策を発見する。

**主要な対立ペア:**

| 対立軸 | ペルソナA | ペルソナB | 論点 |
|--------|-----------|-----------|------|
| 収入 vs 成長 | 経済自立型 A1 | キャリア志向型 A2 | 時給と経験のバランス |
| 育成 vs 即戦力 | 積極投資型 B1 | 即戦力志向型 B3 | 研修の深さ・長さ |
| デジタル vs アナログ | デジタル推進型 D1 | アナログ効率型 D2 | システム導入の是非 |

詳細は [13_ペルソナ分析_手法.md](13_ペルソナ分析_手法.md) を参照。

## 10. 将来の拡張性（Roadmap）

### 10.1 インターンシップ単位認定への接続

- **現状:** 本プロジェクト単体では「単位認定」は行わない（契約・評価の手間が過大であるため）
- **将来構想:** 「自由応募型インターンシップ」の枠組みを活用し、学生個人の申請ベースでの単位化を目指す

### 10.2 運営の学生主体化（Student Secretariat）

- **将来構想:** 事務局業務（データチェック・掲示更新）を、学生スタッフ（RA/TA）に委譲
- これにより職員の負担をゼロに近づけ、学生による学生のためのプラットフォーム運営を実現
